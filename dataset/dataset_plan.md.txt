# Dataset Plan

## 1. Purpose
Цель создания датасета CVE для последующего обучения и оценки интеллектуального ассистента для анализа киберугроз.

## 2. Data Sources
- National Vulnerability Database (NVD)
- CISA Known Exploited Vulnerabilities (KEV)
### 2.1 CWE taxonomy (иерархия)
Для получения полей иерархии CWE (parent/top/depth/path) используется официальная выгрузка MITRE CWE List (XML). 
Иерархия применяется для постановки задачи иерархической классификации (coarse→fine): сначала предсказывается верхний уровень CWE, затем уточнение до конкретного CWE.

## 3. Selection Criteria
- CVE за 2024–2026 годы
- Все CVE из KEV
- Дополнительно случайная выборка CVE из NVD

## 4. Data Fields

### 4.1 Основные поля CVE
- cve_id (string) — идентификатор CVE
- description (string) — описание уязвимости
- published_date (string, YYYY-MM-DD) — дата публикации (для KEV можно использовать dateAdded как прокси)
- severity (string | null) — оценка критичности (если отсутствует в источнике, допускается null)
- cwe (array[string] | null) — список CWE (например ["CWE-79"])
- references (array[string]) — ссылки на источники (минимум ссылка на NVD)
- source (string) — источник записи ("KEV" | "NVD")

### 4.2 Поля CWE-иерархии (для иерархической классификации)
Добавляются для постановки задачи coarse→fine:
- cwe_primary_id (string | null) — основной CWE (если cwe несколько)
- cwe_parent_id (string | null) — родитель CWE в иерархии
- cwe_top_id (string | null) — верхнеуровневая категория (корень цепочки)
- cwe_depth (int | null) — глубина CWE в дереве
- cwe_path (string | null) — путь вида "CWE-1000 > CWE-xxx > CWE-yyy"
- cwe_is_leaf (bool | null) — листовая ли нода (по желанию)

## 5. Format
JSON Lines (JSONL), одна запись — один CVE.

## 6. Limitations
- Возможны неполные описания
- CVSS не всегда отражает реальную эксплуатируемость
- Смещение в сторону публично раскрытых уязвимостей
